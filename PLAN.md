## **План выполнения тестового задания LeadKeeper**

### **1. Выбрать технический вариант**

Для этого задания лучше выбрать **Вариант A: Fullstack**, потому что он сильнее совпадает с вакансией:
- ! SSG SSR
- frontend: **React + TypeScript**
- backend: **Python / FastAPI**
- database: **SQLite**
- ORM: **SQLAlchemy**
- запуск: **Makefile**

Это хорошо попадает в требования компании: frontend-задачи, формы, HTML/CSS/JS/React, простые FastAPI/Python/SQLAlchemy задачи, работа с конфигами и локальными сборками.

---
## **2. Цель прототипа**
Сделать мини-приложение **LeadKeeper**, где:

1. Пользователь заполняет форму заявки.
2. Форма валидируется на клиенте и сервере.
3. Заявка сохраняется в SQLite.
4. Пользователь видит понятное сообщение об успешной отправке.
5. Менеджер может открыть служебный экран и посмотреть список заявок.
6. Проект запускается одной командой через `make`.

## **4. Backend-план**

### **4.1. FastAPI-приложение**
- ! ОЧЕНЬ ВАЖНО ЧТОБЫ ПОТОМ ВЕРСИОНОСТЬ БЫЛА
Сделать API:
```txt
POST /api/leads
GET  /api/leads
GET  /api/health
```
Назначение:
```txt
POST /api/leads
```
Создаёт новую заявку.
```txt
GET /api/leads
```
Возвращает список заявок для служебного экрана.
```txt
GET /api/health
```
Позволяет быстро проверить, что backend запущен.

---
### **4.2. Модель заявки**

Поля:
```txt
id
name
contact
company
comment
consent
created_at
```

Где:
```txt
name      — обязательное поле
contact   — обязательное поле, email или телефон
company   — необязательное поле
comment   — необязательное поле
consent   — обязательное согласие, должно быть true
created_at — дата создания заявки
```

---
### **4.3. Валидация на backend**
Проверить:
```txt
name не пустой
contact не пустой
consent === true
contact похож на email или телефон
```

Важно: даже если есть frontend-валидация, backend всё равно должен проверять данные. Это можно указать в README как базовый production-подход.

---
### **4.4. Хранение**
Использовать SQLite:
```txt
backend/leadkeeper.db
```
Почему SQLite:
- быстро запускается;
- не нужен Docker;
- подходит для тестового прототипа;
- легко проверить локально;
- можно потом заменить на PostgreSQL.

## **5. Frontend-план**
### **5.1. Главная страница с формой**
Страница:
```txt
/
```
или:
```txt
/leads
```
Форма содержит:
```txt
Имя
Телефон или email
Компания
Комментарий
Согласие на обработку данных
Кнопка отправки
```

---
### **5.2. Состояния формы**
Нужно реализовать понятные состояния:
```txt
idle       — форма ещё не отправлялась
loading    — идёт отправка
success    — заявка успешно отправлена
error      — ошибка отправки или валидации
```

После успешной отправки:
```txt
Спасибо! Заявка отправлена. Менеджер свяжется с вами.
```

---
### **5.3. Валидация на frontend**
Минимально:
```txt
Имя обязательно
Телефон или email обязателен
Согласие обязательно
```

Желательно:

```txt
Контакт должен быть похож на email или телефон
```

Например:

- email: `admin@d9911.org`
- телефон: `+7 918 000 000`

---
### **5.4. Админский экран**
Страница:
```txt
/admin
```
На странице показать таблицу:
```txt
Дата
Имя
Контакт
Компания
Комментарий
```
Также можно добавить простые состояния:
```txt
loading
empty
error
```
Если заявок нет:
```txt
Пока нет заявок.
```

---
## **6. Дизайн и адаптивность**
Не делать сложный дизайн. Достаточно аккуратного интерфейса:
```txt
карточка формы
понятные поля
ошибки под полями
адаптивная ширина
таблица на desktop
карточки заявок на mobile
```
Цель — показать, что решение можно быстро вставить на клиентский сайт или лендинг.

---
## **7. Makefile**
Сделать команды:
```makefile
install
dev
backend
frontend
clean
```
Пример логики:
```txt
make install   — установить зависимости frontend и backend
make backend   — запустить FastAPI
make frontend  — запустить Vite
make dev       — запустить backend и frontend
make clean     — удалить временные файлы, кэш, базу при необходимости
```

Для тестового задания удобно, чтобы проверяющий мог сделать:
```bash
make install
make dev
```

И сразу открыть приложение.

## **12. Важное позиционирование под вакансию**
В этом задании лучше показать не «сложный fullstack», а то, что ты умеешь делать клиентские внедрения:
```txt
быстро понять бизнес-сценарий
собрать рабочую форму
подключить backend
сохранить данные
дать менеджеру просмотр
написать понятный README
обозначить ограничения
не усложнить задачу
```
Поэтому не стоит добавлять:
```txt
Docker Compose
JWT
сложную Clean Architecture
Redux
большую UI-библиотеку
микросервисы
сложные роли
```

Если времени мало, лучше простой рабочий результат, чем большая конструкция, которая выглядит тяжело для задачи на 2–4 часа.
